# Codebase To Docs

将现有代码库转化为“可阅读、可核验、可维护、可演进”的文档体系。默认目标不是堆结构图或证据路径，而是先讲清系统在做什么、为什么这样设计、如何运行、如何失败、如何恢复，再用证据回溯这些结论。

## 目标与总纲

生成文档时，始终遵守以下总纲：

- 正文必须以连续可阅读的解释性内容为主体，证据索引为支撑。
- 默认采用“详细解析型输出”，而不是浅层摘要型输出。
- 文档体系必须同时覆盖两条主轴：
  - `架构解析主轴`
  - `工作流解析主轴`
- 先规划页面树，再生成正文，再补 Mermaid 图与证据索引。
- 先讲背景、边界、流程、机制、异常、取舍与风险，再放源码路径、符号和配置键。
- 禁止输出“全是占位符 / 全是引用路径 / 全是表格”的伪深度文档。

### 内容优先原则（强制）

- 主体页面以“可顺序阅读的完整内容”为主，证据引用为辅。
- 正文先回答：`是什么`、`为什么`、`如何运行`、`如何失败`、`如何验证`。
- 证据索引统一后置到章节末或附录，避免打断阅读。
- 图必须服务于解释，而不是充当装饰。
- 没有被正文解释的结论，哪怕引用很多，也视为不合格结论。

### 标题命名规范（强制）

- 用业务语义命名，不用模板语气。
  - 好：`请求生命周期`、`失败模型`、`核心工作流`、`配置面`
  - 差：`系统介绍`、`其他说明`、`补充内容`
- 标题层级不超过 3 级（`#` 到 `###`）。
- 相邻标题不得表达重复含义。
- 标题必须帮助读者判断“这一页解决什么问题”。

## 何时使用

当用户提出以下诉求时使用：

- “分析这个仓库并生成架构文档”
- “从源码补齐用户文档 / 开发文档”
- “现有文档太浅，提升深度”
- “反向梳理模块边界、请求链路、异常链路”
- “把源码转成可维护的知识库”

## 输入参数

执行前确认：

1. 项目路径（repo 根目录）
2. 文档语言（默认中文，除非用户明确要求英文）
3. 范围（完整 / 仅架构 / 仅工作流 / 仅模块 / 仅参考）
4. 深度级别（L1 / L2 / L3，默认 L2）

若用户没有指定，默认使用：

- 语言：中文
- 范围：完整详细解析
- 深度：L2

## 输出模式与主轴

### 默认输出模式：详细解析型

详细解析型输出不是“把代码结构列出来”，而是建立一套解释系统：

- 系统解决什么问题
- 系统由哪些关键部分组成
- 请求和业务流程如何穿过系统
- 失败如何传播、如何恢复
- 为什么采用当前设计
- 哪些地方存在风险与技术债

### 双主轴并行（强制）

详细解析型输出必须同时覆盖以下两条主轴：

1. `架构解析主轴`
   - 上下文与边界
   - 容器 / 组件关系
   - 请求生命周期
   - 失败模型
   - 部署与运维
   - 横切关注点
   - 架构决策
   - 风险与技术债

2. `工作流解析主轴`
   - 工作流地图
   - 核心业务闭环
   - 单条工作流深挖
   - 状态流转
   - 异常与恢复
   - 排障路径
   - 回滚与补偿

### 其他页面类型

除了详细解析型页面，还可以按需补充：

- `任务指南文档（How-to）`
- `参考手册文档（Reference）`
- `原理说明文档（Explanation）`

但这些页面不能替代双主轴的核心深挖页面。

## 页面树规划

### 生成前必须先做页面树规划

在写正文前，必须先定义页面树。至少列出：

- 页面路径
- 页面类型
- 页面目标
- 关键流程 / 关键机制
- 是否需要 Mermaid 图
- 是否需要作为长文深挖页

禁止边写边发散，禁止先拼正文后补结构。

### 推荐页面树

```text
docs/
├── index.md
├── overview/
│   ├── architecture-at-a-glance.md
│   ├── workflow-map.md
│   └── reading-guide.md
├── architecture/
│   ├── system-context.md
│   ├── container-and-component-view.md
│   ├── request-lifecycle.md
│   ├── failure-model.md
│   ├── deployment-and-operations.md
│   ├── cross-cutting-concerns.md
│   ├── design-decisions.md
│   └── risks-and-tech-debt.md
├── workflows/
│   ├── core-business-flows.md
│   ├── workflow-<name>-deep-dive.md
│   ├── state-transitions.md
│   ├── exception-and-recovery.md
│   ├── troubleshooting-playbook.md
│   └── rollback-and-compensation.md
├── modules/
│   ├── module-<name>.md
│   └── module-<name>-runtime.md
├── reference/
│   ├── api.md
│   ├── config.md
│   ├── data-models.md
│   └── glossary.md
└── appendix/
    ├── evidence-index.md
    ├── assumptions.md
    └── verification-notes.md
```

### 详细解析型最小页面集（强制）

若范围为完整详细解析，至少必须有：

- `index.md`
- `overview/architecture-at-a-glance.md`
- `overview/workflow-map.md`
- `architecture/request-lifecycle.md`
- `architecture/failure-model.md`
- `workflows/core-business-flows.md`
- `workflows/exception-and-recovery.md`
- `reference/config.md`
- `appendix/evidence-index.md`

### 页面职责分工

- `index.md`：说明这套文档怎么读、从哪里开始
- `architecture-at-a-glance.md`：给出整体结构与阅读顺序
- `workflow-map.md`：概览关键工作流及其关系
- `request-lifecycle.md`：逐跳解释一条典型请求如何穿过系统
- `failure-model.md`：解释失败触发点、传播路径与恢复方式
- `core-business-flows.md`：解释最关键的业务闭环
- `workflow-<name>-deep-dive.md`：深挖单条工作流的状态、异常、补偿
- `reference/*`：存放配置、API、数据模型等事实性材料
- `appendix/*`：存放证据索引、假设、验证记录

### 长文深挖页定义（强制）

以下页面默认应作为长文深挖页处理：

- `architecture/request-lifecycle.md`
- `architecture/failure-model.md`
- `workflows/workflow-<name>-deep-dive.md`

长文深挖页至少要包含：

- 背景与范围
- 参与者 / 组件
- 正常流逐跳解析
- 异常流逐跳解析
- 状态或数据变化
- 关键约束
- 设计取舍
- 风险与观测点
- 相关页面
- 证据引用或证据索引

## 正文写作规范

### 页面开头固定块

除纯参考页外，每页开头必须包含：

- `摘要`
- `你将了解`
- `范围`

任务指南页可以用 `目标` 替代 `你将了解`。

### 正文必须优先写什么

正文优先展开以下内容：

- 业务背景
- 系统边界
- 核心机制
- 关键流程
- 失败与恢复
- 设计取舍
- 风险与技术债

证据、配置键、函数路径、符号索引等内容只能支撑这些叙述，不能取代这些叙述。

### 正常流与异常流（强制）

每个核心架构页或工作流页，至少包含：

- `flow-normal`：一条正常流逐跳解析
- `flow-exception`：一条异常流逐跳解析

逐跳解析必须写明：

- 调用点
- 被调用点
- 输入
- 输出
- 副作用
- 失败模式
- 恢复策略

### 设计取舍（强制）

每个核心页面至少包含 1 组设计取舍：

- 当前方案是什么
- 替代方案是什么
- 为什么没有选替代方案
- 替代方案的代价是什么（复杂度 / 一致性 / 性能 / 运维）

### 风险与技术债（强制）

风险不能泛写。每个风险项至少包含：

- 触发条件
- 影响范围
- 可观测信号
- 缓解动作

### 证据引用格式

核心结论必须可回溯，引用格式如下：

```text
证据：`path/to/file.ext` -> `SymbolOrKey`（用途说明）
```

示例：

```text
证据：`src/service/order_service.py` -> `OrderService.create_order`（下单主流程）
证据：`config/default.yaml` -> `retry.max_attempts`（重试上限来源）
```

## Mermaid 图谱规范

Mermaid 图必须服务于解释，而不是装饰。

### 图谱分类（强制）

1. `结构图`
   - 展示系统边界、容器、组件、外部依赖
2. `运行时图`
   - 展示请求 / 事件在系统中的流动
3. `工作流图`
   - 展示业务闭环、角色、状态与决策点
4. `故障 / 恢复图`
   - 展示失败触发、传播、降级、恢复与人工介入

### 每张图都必须有图后解释

每张 Mermaid 图后都必须紧跟解释文字，至少说明：

- 这张图表示什么
- 应该按什么顺序阅读
- 哪几个节点最关键
- 哪些细节在正文中继续展开

没有图后解释的 Mermaid 图，视为无效图。

### 图与正文的顺序

推荐顺序：

1. 先用 1 段文字说明这一节要解释什么
2. 放 Mermaid 图
3. 写读图说明
4. 展开逐跳解析或机制拆解
5. 最后补证据索引

### Mermaid 图数量与覆盖预算

#### L1

- 至少 1 张结构图或工作流图

#### L2

- 至少 1 张结构图
- 至少 1 张运行时图或工作流图
- 至少 1 条图后读图说明
- 至少 2 条完整流程叙述（正常 + 异常）

#### L3

- 至少 1 张结构图
- 至少 1 张运行时图
- 至少 1 张工作流图
- 至少 1 张故障 / 恢复图
- 每张图都必须有读图说明
- 至少 2 张图被正文逐跳展开

## 生成工作流

复制此清单并逐项完成：

```text
Task Progress:
- [ ] 1. 深度校准（范围 + L 级别 + 输出语言）
- [ ] 2. 页面树规划（主轴 + 页面职责 + 深挖页）
- [ ] 3. 证据索引（文件 / 符号 / 配置 / 测试）
- [ ] 4. 系统建模（结构层 / 行为层 / 运行层）
- [ ] 5. 内容生成（先正文，再图，再附录证据）
- [ ] 6. 四阶段验证（完整性 / 一致性 / 深度性 / 可读性）
- [ ] 7. 问题修复与最终汇总
```

### 深挖执行法（避免“只写表层结构”）

按顺序执行，不可跳步：

1. 先收敛再展开：先画全局结构与工作流地图，再选 2-3 条高价值链路深挖。
2. 逐跳拆解调用链：每一跳都写明输入 / 输出 / 副作用 / 失败模式 / 恢复策略。
3. 反例驱动补全：对关键结论补一个“何时不成立”的边界条件。
4. 设计取舍落地：每个关键设计至少给一个替代方案，并量化代价。
5. 风险量化：风险必须包含触发条件、影响范围、观测信号、缓解动作。

### 必须产物（每次生成都要有）

- `正文叙述`：背景、边界、关键机制、关键流程的连续说明
- `page-tree`：页面树及页面职责
- `flow-normal`：至少一条正常流逐跳解析
- `flow-exception`：至少一条异常流逐跳解析
- `diagram-set`：至少一张 Mermaid 图（按级别增加类型覆盖）
- `diagram-explainer`：每张图后的读图说明
- `decision-log`：至少 3 条设计取舍（L3 为 5 条）
- `risk-register`：风险登记表
- `evidence-index`：附录证据索引

## 验证与失败判定

### 深度级别

| 级别 | 目标 | 最低要求 |
|------|------|----------|
| L1 | 快速交接 | 结构说明 + 关键流程 |
| L2 | 工程维护（默认） | 正常 / 异常链路 + 设计取舍 + 风险 |
| L3 | 审计 / 重构 | L2 + 故障模型 + 恢复 / 降级 + 容量或演进分析 |

### 内容密度预算（强制）

未达到以下预算，判定“内容深度不足”：

说明：

- 本表用于校验“正文有没有真正展开讲透”，重点看连续叙述、关键流程、图后解释和恢复路径。
- 本表适用于 `深挖页` 与 `核心架构 / 工作流 / 模块页`。
- `index.md`、纯 `reference/*`、纯 `appendix/*` 页面不适用本表，应按导航性、覆盖性与可追溯性校验。

| 文档类型 | L1 | L2 | L3 |
|---------|----|----|----|
| 架构文档 | >=1 张结构图或运行时图，>=1 条主链路叙述 | >=1 张结构图 + >=1 张运行时图，>=1 条正常流逐跳解析，>=1 条异常流逐跳解析，>=1 组图后解释 | >=3 类图覆盖，>=1 条正常流，>=1 条异常流，>=1 条恢复/降级链路，>=2 组图后解释 |
| 工作流文档 | >=1 张工作流图，>=1 个成功闭环说明 | >=1 张工作流图或状态图，>=1 个成功闭环，>=1 个失败闭环，>=1 条恢复/补偿说明 | >=2 张图，>=2 个成功闭环，>=1 个失败闭环，>=1 条回滚路径，>=1 条人工介入或补偿路径 |
| 模块文档 | >=1 张图，>=1 个主流程机制说明 | >=1 张运行时图，>=1 个正常流，>=1 个异常流，>=2 个关键机制说明 | >=2 张图，>=2 个流程（含并发或重试之一），>=3 个关键机制说明，>=1 条容量或并发场景说明 |

### 深度预算（强制）

说明：

- 本表用于校验 `深挖页` 与 `核心架构 / 工作流 / 模块页` 的深度。
- `index.md`、纯 `reference/*`、纯 `appendix/*` 页面不适用本表，应按覆盖性与可追溯性校验。
- `内容密度预算` 检查“有没有展开讲透”，本表检查“讲透之后是否达到可维护深度”。

| 文档类型 | L1 | L2 | L3 |
|---------|----|----|----|
| 架构文档 | >=6 条证据，>=1 条调用链，>=1 组设计取舍，>=1 组风险 | >=12 条证据，>=2 条调用链（正常+异常），>=3 条设计取舍，>=3 条风险 | >=18 条证据，>=3 条调用链（含恢复/降级），>=5 条设计取舍，>=5 条风险，>=1 个容量或演进分析 |
| 工作流文档 | >=4 条证据，>=1 个成功闭环，>=1 个失败分支 | >=8 条证据，>=2 个闭环（成功+失败），>=1 条恢复/补偿路径，>=3 条排障项 | >=12 条证据，>=3 个闭环（含回滚），>=2 条恢复/补偿路径，>=5 条排障项，>=1 条人工介入路径 |
| 模块文档 | >=5 条证据，>=1 个主流程，>=1 个关键机制，>=1 组风险 | >=10 条证据，>=2 个流程（正常+异常），>=2 个关键机制，>=3 条风险，>=1 组设计取舍 | >=14 条证据，>=3 个流程（含并发或重试），>=3 个关键机制，>=5 条风险，>=2 组设计取舍，>=1 个并发或容量场景 |

### 四阶段验证

1. `完整性`
   - 是否有页面树
   - 是否覆盖架构主轴和工作流主轴
   - 是否有正常流与异常流
   - 是否有设计取舍
   - 是否有风险与技术债
   - 是否有证据索引
   - 是否有 Mermaid 图

2. `一致性`
   - 术语是否一致
   - 图与正文是否一致
   - 结论与证据是否一致
   - 页面职责是否混乱
   - 正常流与异常流是否矛盾

3. `深度性`
   - 是否有连续叙述的背景 / 机制 / 边界
   - 是否逐跳说明调用链
   - 是否覆盖输入 / 输出 / 副作用 / 失败模式
   - 是否有反例或边界条件
   - 是否有替代方案与代价
   - 是否有结构化风险分析

4. `可读性`
   - 页面目标是否清晰
   - 是否大量堆砌路径 / 符号 / 清单
   - 图后是否有解释
   - 阅读顺序是否清晰
   - 是否有 `相关页面`

### 失败判定（直接返工）

出现任一情况，当前文档视为不合格：

- 缺“异常流”章节
- 缺“设计取舍”章节
- 缺“风险与技术债”章节
- 缺 Mermaid 图，或图后没有解释
- 主文以证据列表、路径列表、占位条目为主
- 调用链只写步骤，不写失败模式或副作用
- 只有架构页，没有工作流页
- 只有总览 / 参考页，没有深挖页
- 风险项缺触发条件或可观测信号
- 页面之间没有导航关系
- 存在 3 个及以上“无证据结论”

## 限制

- 动态注入、运行时反射、AOP、代码生成行为可能需要人工补充验证。
- 超大型代码库（1000+ 模块）应按子域分批生成，而不是一次性覆盖全仓。
- 文档中的密钥、凭证、租户信息、脱敏规则需人工二次审查。
- 若源码本身存在未实现分支或死路径，应显式标注为“待验证假设”，不得伪装成已确认行为。
