# Validation Reference

## 四阶段验证

### 1) 完整性

检查是否覆盖了详细解析型文档的最小骨架：

- 是否先定义了页面树
- 是否同时覆盖 `架构解析主轴` 与 `工作流解析主轴`
- 是否包含至少一条正常流
- 是否包含至少一条异常流
- 是否包含设计取舍
- 是否包含风险与技术债
- 是否包含证据索引
- 是否包含 Mermaid 图

若缺任意核心块，直接判定不通过。

### 2) 一致性

检查文档、图与源码是否一致：

- 类名 / 函数名 / 配置键 / 常量值一致
- 图中节点命名与正文术语一致
- 文档中的路径真实存在
- 关键结论可被证据支持
- 页面职责没有混写
- 正常流与异常流不互相矛盾

执行建议：

- 使用 `rg` 校验符号、路径、配置键
- 发现冲突时，以源码为准修正文档

### 3) 深度性

检查文档是否真的在解释系统，而不是只罗列材料：

- 有连续叙述的背景、流程、机制、边界
- 至少一条调用链被逐跳拆解
- 每一跳写明输入 / 输出 / 副作用 / 失败模式 / 恢复策略
- 至少一处说明边界条件或反例
- 至少一组设计取舍包含替代方案与代价
- 风险分析结构化，包含触发条件 / 影响范围 / 可观测信号 / 缓解动作
- Mermaid 图被正文解释，而不是单独摆放

### 4) 可读性

检查文档是否真能被顺序阅读：

- 页面开头包含 `摘要` 与 `你将了解`（任务指南可用 `目标`）
- 标题语义化且简短
- 没有 TODO / TBD / FIXME / `...` 占位
- 代码块语言标注率 100%
- 图后紧跟读图说明
- 没有路径 / 符号 / 证据列表主导正文
- 页面末尾包含 `相关页面`

## 内容密度建议

说明：

- 本节用于辅助判断“正文是否真正展开讲透”，与 `SKILL.md` 中的 `内容密度预算（强制）` 保持同一口径。
- `index.md`、纯 `reference/*`、纯 `appendix/*` 页面不适用这些建议。

### L2 架构页最低建议

- 2 段以上连续正文解释背景与边界
- 1 条正常流逐跳解析
- 1 条异常流逐跳解析
- 1 张结构图 + 1 张运行时图，并有图后解释
- 1 组设计取舍
- 1 组风险项

### L2 工作流页最低建议

- 1 个成功闭环
- 1 个失败闭环
- 1 张工作流图或状态图
- 1 条恢复 / 补偿说明
- 1 组排障路径

### L3 加严建议

- 增加恢复 / 降级场景
- 增加回滚或补偿路径
- 增加容量、并发或时序压力场景
- 增加至少一条人工介入路径

## Mermaid 质量门槛

Mermaid 图必须满足以下要求：

- 图有明确用途：结构、运行时、工作流、故障 / 恢复之一
- 图后有读图说明
- 图中的节点能在正文里找到解释
- 图没有和正文术语冲突
- 图不是唯一说明方式，正文必须继续展开

以下情况视为 Mermaid 不合格：

- 只有图，没有图后解释
- 图后解释只复述标题，没有说明阅读顺序
- 图中的关键分支没有被正文展开
- 一页多张图，但没有说明各图分工

## Severity Levels

- `🔴 严重`：事实错误、关键缺失、误导性描述，必须修复
- `🟡 警告`：可读性或完整性不足，建议修复
- `🟢 通过`：达到当前深度级别要求

## 返工规则

以下任一命中，直接判定不通过并返工：

- 缺异常流章节
- 缺设计取舍章节
- 缺风险与技术债章节
- 缺 Mermaid 图或图后无解释
- 主文缺少流程与机制展开，仅有列表或引用
- 只有架构页，没有工作流页
- 只有总览页，没有深挖页
- 调用链缺逐跳字段
- 风险项缺触发条件或可观测信号
- 文档描述与源码冲突
